# 일일 리포트 생성 루틴

## 루틴 이름

일일 투자 리포트 생성

## 실행 트리거

사용자가 다음과 같이 요청하면 이 루틴을 실행한다.

- "오늘 리포트 뽑아줘"
- "데일리 리포트 작성"
- "일일 리포트 만들어줘"
- "루틴 돌려줘"
- "장마감 리포트"

## 실행 시점

- 기본 기준일: 현재 한국시간 날짜
- 기본 작성 시점: 한국장 마감 이후
- 해외주식은 전일 또는 현재 확인 가능한 NASDAQ 데이터를 함께 기록한다.
- 휴장일이거나 데이터가 비정상적이면 리포트에 `데이터 품질 메모`로 남긴다.

## 입력 파일

- 기본 관찰종목 예시: `_report/di/watchlist.example.yaml`
- 개인 관찰종목: `_report/di/watchlist.yaml`
- 리포트 템플릿: `_report/di/templates/daily-report.md`
- 결정 로그: `_report/di/decisions/decision-log.md`
- 시장 환경 루틴: `_report/quant/routines/market-regime-scan-routine.md`

`_report/di/watchlist.yaml`이 있으면 우선 사용한다. 없으면 `_report/di/watchlist.example.yaml`을 기준으로 삼는다.

## 기본 관찰종목

현재 루틴의 기본 관찰종목은 다음과 같다.

| 종목코드 | 종목명 | 시장 | 상태 |
| --- | --- | --- | --- |
| 005930 | 삼성전자 | KRX | 관찰중 |
| 000660 | SK하이닉스 | KRX | 관찰중 |
| NAS:NVDA | NVIDIA | NASDAQ | 관찰중 |
| 454910 | 두산로보틱스 | KRX | 관찰중 |

종목이 추가되면 `_report/di/watchlist.yaml`에 먼저 반영하고, 리포트의 `관찰종목` 메타데이터에도 포함한다.

## MCP 조회 순서

국내주식은 API 호출 전에 반드시 `find_api_detail`로 파라미터를 확인한다.

1. 종목코드 확인
   - `domestic_stock.find_stock_code`
2. 현재가
   - `domestic_stock.inquire_price`
3. 일봉
   - `domestic_stock.inquire_daily_itemchartprice`
   - 기본 조회 범위: 기준일 이전 60일
   - 리포트에는 최근 10거래일 내외의 가격 위치를 확인하되, 단순 종가 차트보다 상대수익률·거래량 배율·고점 대비 낙폭처럼 판단에 직접 연결되는 그래프를 우선한다.
4. 투자자 수급
   - `domestic_stock.investor_trade_by_stock_daily`
   - 수급이 판단 근거일 때는 단순 일별 막대보다 외국인·기관 누적 순매수 그래프를 우선하고, 필요하면 일별 막대를 보조로 사용한다.
5. 뉴스/공시 제목
   - `domestic_stock.news_title`
   - 표준 파라미터가 확정되지 않았으면 무리하게 호출하지 말고 `미수집`으로 기록한다.

해외주식은 다음 API부터 사용한다.

1. 현재체결가
   - `overseas_stock.price`
2. 기간별시세
   - `overseas_stock.dailyprice`

## 리포트 작성 절차

1. 기준일을 확정한다.
2. 관찰종목 목록을 확인한다.
3. 각 API의 상세 파라미터를 확인한다.
4. 필요하면 `_report/quant/routines/market-regime-scan-routine.md`로 Market Regime을 확인하고 `시장 배경`에 한 줄 요약만 반영한다.
5. 종목별 데이터를 조회한다.
6. 종목별 가격·거래량·수급 중 판단에 중요한 항목을 그래프로 만든다.
7. `_report/di/daily/YYYY/YYYY-MM-DD.md`를 작성한다.
8. 종목별 판단을 `_report/di/decisions/decision-log.md`에 추가한다.
9. 누락 데이터, API 실패, 종목명 불확실성을 `데이터 품질 메모`에 적는다.
10. 사용자가 요청하지 않는 한 주문 API는 호출하지 않는다.

## 설명과 근거 작성 규칙

- 전문용어가 나오면 처음 등장한 위치나 바로 다음 문장에서 짧게 풀어서 설명한다. 예: `MLCC(전기를 안정적으로 공급하는 적층 세라믹 콘덴서)`.
- 사실 주장은 근거를 함께 적는다. 공식 자료, 공시, IR 자료, 기업 보도자료, 거래소/감독기관 자료, 1차 보도를 우선 사용한다.
- 위키, 커뮤니티, 블로그 요약글은 판단 근거로 사용하지 않는다. 보조 아이디어로 보더라도 리포트의 근거 출처에는 넣지 않는다.
- 확인하지 못한 내용은 `확실하지 않음` 또는 `미확인`으로 표시하고, 추정치를 사실처럼 쓰지 않는다.
- 종목별 해석에는 긍정적 측면, 부정적 측면, 주요 리스크를 항상 함께 적는다.
- 리포트나 종목 분석의 끝에는 사용자가 다음에 이어서 물어볼 만한 후속 질문 3개를 제안한다.

## 그래프 작성 규칙

- 기본 형식은 Obsidian Charts 플러그인의 Markdown `chart` 코드블록을 사용한다.
- `chart` 코드블록은 YAML 문법으로 작성하고, 최소 `type`, `labels`, `series`를 둔다.
- 이미지 차트(PNG/JPG/SVG 파일)나 외부 HTML 차트는 사용자가 명시적으로 요청한 경우에만 만든다.
- Mermaid `xychart-beta`는 사용하지 않는다. Charts 플러그인을 사용할 수 없으면 차트 대신 표와 해석으로 남긴다.
- 각 정상 관찰종목에는 최소 1개 이상의 판단형 `chart` 코드블록을 둔다. 단순 종가 차트는 가격 기준선 설명에 꼭 필요할 때만 사용한다.
- 같은 그룹 안에서 여러 종목을 비교할 때는 절대가격 축을 억지로 맞추지 말고 기준일을 100으로 둔 상대지수 표와 `chart` 코드블록을 추가한다.
- 같은 그룹 비교용 `chart` 코드블록은 범례, 공통 x축, 공통 y축 범위를 갖게 만든다. 예: `legendPosition: bottom`, `yMin`, `yMax`.
- 상대지수 표 또는 차트 아래에는 `100=기준일 종가`, `110=기준일 대비 +10%`, `95=기준일 대비 -5%`처럼 읽는 법을 적는다.
- 유의미한 판단형 차트 후보는 다음 순서로 검토한다.
  - 같은 그룹 상대수익률: 기준일 종가를 100으로 둔 상대지수.
  - 누적 수급: 외국인+기관 또는 판단에 필요한 주체의 누적 순매수.
  - 거래량 배율: 전일 거래량 대비 당일 거래량 비율.
  - 최근 고점 대비 낙폭: 조회 구간 최고가 대비 종가 하락률.
- 가격만으로 판단하기 어려운 국내 종목은 누적 수급 그래프를 우선 사용하고, 당일 주체별 충돌이 중요할 때만 일별 순매수 `bar` 차트를 보조로 추가한다.
- 정확한 종목코드가 확정되지 않은 항목은 확정 종목처럼 차트를 만들지 말고 후보 비교 또는 데이터 한계 메모로 표시한다.
- 표 또는 차트 아래에는 반드시 해당 데이터를 참조한 해석을 1개 이상 적는다.
- 단위는 표 머리글 또는 차트 축에 명확히 적는다. 예: 원, 만원, 달러, 백만 주, 지수. 국내 가격 비교에는 기본적으로 `만원` 또는 상대지수 `지수`를 사용한다.

## 판단 표현 규칙

판단값은 한국어로 통일한다.

- 관망
- 매수 후보
- 추가 확인
- 비중 축소 후보
- 매도 후보
- 종목 확인

각 판단에는 반드시 다음을 붙인다.

- 근거
- 실행 조건
- 무효화 조건
- 재검토일

## 저장 규칙

- 사람이 읽는 리포트: `_report/di/daily/YYYY/YYYY-MM-DD.md`
- 결정 누적: `_report/di/decisions/decision-log.md`
- 원천 JSON: `_report/raw/YYYY/YYYY-MM-DD/SYMBOL/`

현재는 수동 루틴에서 원천 JSON을 기본 저장하지 않는다. 저장하지 않은 경우 리포트에 `조회 완료, 파일 미저장`으로 남긴다.

## 완료 기준

루틴은 다음이 완료되면 끝난다.

- 기준일 일일 리포트가 생성 또는 갱신됨
- 관찰종목별 그래프와 그래프 기반 해석이 있음
- 관찰종목별 판단이 있음
- 결정 로그가 갱신됨
- 누락 데이터와 확인 필요 항목이 명시됨
- 실제 주문 API를 호출하지 않았음
